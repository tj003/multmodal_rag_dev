Part 8: The WHERE Clause - Filter 1

WHERE
    dc.document_id = ANY(filter_document_ids)

What it does: Only include chunks from the specified documents

Breaking it down:
- filter_document_ids is the array of UUIDs you passed as a parameter (e.g., ['doc-123', 'doc-456'])
- ANY(filter_document_ids) means "any value inside this array"
- dc.document_id = ANY(...) means "the chunk's document_id matches at least one UUID in the array"

Part 9: The WHERE Clause - Filter 2

AND dc.embedding IS NOT NULL

What it does: Skip chunks that don't have embeddings yet

Part 10: The WHERE Clause - Filter 3 (The Core Vector Search)

AND (1 - (dc.embedding <=> query_embedding)) > match_threshold

What it does: Only keep chunks that are similar enough to the query

Step 1: Calculate cosine distance
dc.embedding <=> query_embedding

- <=> = cosine distance operator (pgvector)
- Calculates how "distant" the chunk's embedding is from the query embedding
- Lower value = more similar (vectors are closer)

Step 2: Convert cosine distance to cosine similarity
1 - (dc.embedding <=> query_embedding)

- Converts cosine distance into a cosine similarity score
- Higher value = more similar
- Range: approximately 0.0 (totally different) to 1.0 (identical)

Example:
If distance = 0.2 → similarity = 1 - 0.2 = 0.8 (80% similar) ✅
If distance = 0.8 → similarity = 1 - 0.8 = 0.2 (20% similar) ❌

Step 3: Apply threshold

- Only keeps chunks with similarity ≥ threshold (default 0.3 = 30%)
- Filters out irrelevant results

Why: Without this, you'd get millions of results, most irrelevant. This keeps only semantically similar content.

Part 11: The ORDER BY Clause

ORDER BY
    dc.embedding <=> query_embedding ASC

What it does: Sorts the remaining chunks by similarity (most similar first)

Breaking it down:
- Uses the same <=> operator to calculate distance
- Smaller distance = more similar = appears first in results
- This ensures the best matches are at the top

NOTE!

We use the distance calculation twice:
1. In WHERE to filter (must be above threshold)
2. In ORDER BY to rank (best matches first)

This is intentional and efficient - PostgreSQL can use the HNSW index for both operations.
